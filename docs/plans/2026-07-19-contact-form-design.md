# 聯絡表單設計

## 目標

將「暖心後盾」頁面的「聯絡我們」分頁改成可直接送出的聯絡表單。表單內容沿用舊站 `https://beunion.tw/contact/` 的主要欄位，並將通知寄送至 `volt02332@gmail.com`。

## 使用者介面

- 延續現有網站的深藍描邊、圓角卡片及暖色系視覺。
- 桌機版採表單與聯絡資訊雙欄配置，行動版改為單欄。
- 欄位包含：
  - 姓名（必填）
  - Email（必填）
  - 手機號碼（必填）
  - 目前從事的工作／聯絡內容（選填）
- 提供清楚的必填提示、送出中狀態、成功訊息及失敗訊息。
- 成功送出後清空表單；送出期間停用按鈕，避免重複提交。
- 表單下方保留電話與聯絡信箱資訊。

## 架構與資料流程

1. `ShieldHub` 中的聯絡分頁顯示受控表單。
2. 瀏覽器以 `POST /api/contact` 傳送 JSON。
3. API 在伺服器端重新驗證、去除欄位前後空白，並拒絕格式錯誤或超長資料。
4. API 使用部署平台的 `RESEND_API_KEY` 呼叫 Resend REST API。
5. 通知寄至 `volt02332@gmail.com`，並將訪客 Email 設為 `Reply-To`，方便管理員直接回覆。
6. API 僅回傳一般化的成功或錯誤訊息，不向瀏覽器暴露寄信服務細節或金鑰。

## 防護措施

- Resend 金鑰只從伺服器環境變數讀取，不寫入前端或原始碼。
- 伺服器限制請求方法與 `Content-Type`，並限制請求本文大小。
- 姓名、Email、手機及內容均設定長度與格式限制。
- 加入視覺隱藏的 honeypot 欄位；自動填入該欄位的請求直接回覆成功但不寄信。
- 郵件內容使用純文字，避免把訪客輸入當成 HTML 執行。
- 不將聯絡內容儲存到瀏覽器或資料庫；本次範圍只寄送通知。

## 錯誤處理

- 欄位不合法：回傳 `400` 與可理解的表單提示。
- 請求格式或方法不正確：回傳 `405` 或 `415`。
- Resend 未設定或寄送失敗：伺服器記錄不含敏感資料的錯誤，前端顯示稍後再試與備用聯絡信箱。
- 成功寄送：回傳 `200`，顯示完成訊息。

## 測試與驗證

- 單元測試涵蓋有效資料、缺少必填欄位、Email 格式、長度限制與 honeypot。
- API 測試涵蓋成功寄送、Resend 失敗、缺少環境變數及不支援的方法。
- 元件來源測試確認舊站欄位、API 路徑、送出狀態與目的信箱資訊存在。
- 完成後執行完整測試、TypeScript lint、正式版建置及線上表單檢查。
